# PayrollPadi — DATABASE.md

Everything PayrollPadi stores lives in **Cloud Firestore**, Firebase's database. This document explains every collection: what it's for, what fields it has, and how documents relate to each other.

---

## 0. Firestore in 60 Seconds (for readers new to it)

Firestore is a **NoSQL document database**. There are no tables, no rows, no SQL, and — critically — **no foreign keys enforced by the database itself**. Instead:

- Data lives in **collections** (like folders), each containing **documents** (like individual JSON files).
- A document can contain nested objects ("maps") and arrays directly — no need for separate linking tables for simple structures.
- One document can have its own **subcollections** nested inside it (e.g. every payroll run has its own `payslips` subcollection).
- Relationships between documents (e.g. "this payslip belongs to this employee") are just **plain text ID strings stored as a field** — the database does not automatically keep them in sync or stop you from creating a dangling reference. This is a real trade-off: it's fast and flexible, but it means **the application code is responsible for keeping IDs consistent**, and a mismatched ID silently produces "no data found" rather than a database error. (This is exactly what an early debugging session on this project mistakenly suspected as the cause of a permissions bug — see `HANDOVER.md` Section 6 for what it actually turned out to be.)

## 2. `/users/{uid}`

**Purpose:** the login-identity record. One document per Firebase Auth account. This is what the app checks first after login to decide "are you an Admin or an Employee, and if Employee, which staff record is yours?"

The document ID (`{uid}`) is the Firebase Authentication user ID — the same ID Firebase issues automatically when an account is created. This is the one collection where the document ID is *not* auto-generated by Firestore; it's deliberately set to match the Auth UID so the two are always the same value.

| Field | Type | Notes |
|---|---|---|
| `email` | string | |
| `displayName` | string | |
| `role` | `"admin"` \| `"employee"` | **Must be spelled exactly `role`** — a manually-typed `roll` (missing an `e`) silently breaks role checks with no error, since Firestore doesn't validate field names. This happened once during setup; see `HANDOVER.md`. |
| `employeeId` | string \| `null` | Points to a document in `/employees`. `null` for admins. |
| `createdAt` | timestamp | |

**Who can read it:** only the user themselves (`request.auth.uid == uid`). Admins do **not** get blanket read access to other users' `/users` docs directly — they access employee data via `/employees` instead.

**Who can write it:** only an existing admin (checked via `getRole() == "admin"`). Employees can never write their own user doc directly.

## 3. `/employees/{employeeId}`

**Purpose:** the actual staff record — salary, bank details, tax IDs. Document ID is a random Firestore-generated ID (not the same as the Auth UID — the two are linked via the `uid` field below).

| Field | Type | Notes |
|---|---|---|
| `uid` | string | The matching Firebase Auth UID — this is how an employee's `/users/{uid}` doc and their `/employees/{employeeId}` doc are linked together. |
| `name` | string | |
| `email` | string | |
| `department` | string | |
| `employeeIdNumber` | string | A human-readable staff ID (e.g. "0001") — separate from the Firestore document ID, which is a long random string. |
| `bankAccount` | map `{ accountNumber, bankName }` | The **current, approved** bank details. |
| `bankAccountPending` | map \| `null` | A requested change awaiting admin approval. Never applied automatically. |
| `TIN` | string | Tax Identification Number. |
| `pensionRSA` | string | Pension RSA number. |
| `salaryStructure` | map `{ basic, housing, transport, otherAllowances }` | Monthly amounts in ₦. This is the direct input to every payslip calculation — see `CALCULATIONS.md`. |
| `voluntaryDeductions` | array of `{ label, amount }` | Loans, union dues, etc. |
| `status` | `"active"` \| `"inactive"` | Inactive employees are skipped when payroll is run. |
| `createdAt` | timestamp | |

**Who can read it:** an admin (any employee record), or the employee themselves (only their own — checked via their `/users` doc's `employeeId` field matching this document's ID).

**Who can write it:** admins only, always. **Employees can never write to this collection directly, even their own bank details** — that's the entire point of the `bankAccountPending` field and the approval workflow. A direct write attempt from an employee is rejected by `firestore.rules`, not just hidden by the UI.

## 4. `/taxSettings/{versionId}`

**Purpose:** the tax rules currently (or previously) in effect. **Every save creates a brand-new document — nothing is ever edited in place.** This is what makes historical payslips immune to later rate changes (see `CALCULATIONS.md` Section 12).

| Field | Type | Notes |
|---|---|---|
| `bands` | array of `{ min, max, rate }` | The progressive PAYE tax bands. `max: null` means "no upper limit" (only the last band). |
| `pensionRate` | number | Decimal, e.g. `0.08` for 8%. |
| `nhfRate` | number | Decimal. |
| `rentReliefCapAnnual` | number | ₦, per year. |
| `rentReliefPercent` | number | Stored but not currently used in the calculation — see `CALCULATIONS.md` Section 6 for why (documented simplification). |
| `effectiveFrom` | timestamp | Used to determine which version is "current" — the most recent one by this field. |
| `createdBy` | string | Admin UID who saved it. |
| `createdAt` | timestamp | |

**Who can read it:** admins only.

**Who can write it:** admins can **create** new versions. **Nobody can update or delete an existing version, ever** (`allow update, delete: if false` in the rules — this is enforced by Firestore itself, not just the UI).

## 5. `/payrollRuns/{runId}`

**Purpose:** one document per payroll cycle (e.g. "July 2026"). Document ID is a random Firestore-generated ID.

| Field | Type | Notes |
|---|---|---|
| `period` | string | `"2026-07"` format. |
| `status` | `"draft"` \| `"processing"` \| `"locked"` | `draft` = computed but not yet emailed. `locked` = finalized, emails sent, effectively frozen. |
| `taxSettingsVersionUsed` | string | The `/taxSettings/{versionId}` that was current when this run was computed — locked in permanently. |
| `totalCost` | number | Sum of every included employee's gross pay. |
| `employeeCount` | number | How many employees were included. |
| `runBy` | string | Admin UID who triggered the run. |
| `runAt` | timestamp | |
| `lockedAt` | timestamp \| `null` | Set only once the run is locked. |

**Who can read/write it:** admins only.

### 5a. `/payrollRuns/{runId}/payslips/{employeeId}` (subcollection)

**Purpose:** one document per employee, per run — the actual computed payslip. Document ID is set to match the employee's `/employees/{employeeId}` document ID (deliberately, so a payslip and its owning employee always share the same ID).

| Field | Type | Notes |
|---|---|---|
| `employeeId` | string | Duplicated here (matches the document ID) specifically so it can be used as a query filter — see the note on custom claims below. |
| `employeeName`, `employeeIdNumber` | string | Snapshotted at run time, so the payslip still shows correctly even if the employee's name changes later. |
| `period` | string | |
| `runId` | string | |
| `taxSettingsVersionUsed` | string | |
| `grossPay`, `netPay` | number | |
| `earnings` | map `{ basic, housing, transport, otherAllowances }` | |
| `deductions` | map `{ pension, nhf, nhis, payeTax, voluntary, rentReliefApplied }` | |
| `payeBreakdown` | array of `{ bandMin, bandMax, rate, taxOnBand }` | The full band-by-band trace — this is what's rendered as the "PAYE Band Breakdown" table on every payslip. |
| `payslipUrl` | string \| `null` | Cloudinary URL, if configured. `null` otherwise — this is expected, not an error. |
| `emailSentAt` | timestamp \| `null` | Set when the payslip email actually sends. |

**Who can read it:** an admin (any payslip), or the owning employee reading **only their own** payslip.

**⚠️ This is the one rule in the whole system that required a non-obvious fix — worth understanding if you're extending this collection.** The "My Payslips" feature searches *across every run's* `payslips` subcollection at once (a `collectionGroup` query), rather than reading one specific folder. Firestore requires this kind of cross-collection query to be provably safe *before* it even runs — and a rule that needs to look up another document (`get()`) to check ownership can't be proven safe this way. The fix was to check `request.auth.token.employeeId` — a value baked directly into the employee's login token via **Firebase Auth custom claims** — instead of looking it up from Firestore. Full story in `AUTHENTICATION.md` and `HANDOVER.md` Section 6.

**Who can write it:** admins only, and in practice the app never edits a payslip after the run is locked — payslips are effectively immutable financial records once created.

## 6. `/notifications/{uid}/items/{notifId}` (subcollection)

**Purpose:** in-app notifications (the bell icon). One top-level document per user (`{uid}`, matching their Auth UID), with a subcollection of individual notification items.

| Field | Type | Notes |
|---|---|---|
| `type` | `"payslip_ready"` \| `"payroll_locked"` \| `"bank_change_pending"` | |
| `message` | string | |
| `payslipId` | string \| `null` | |
| `read` | boolean | |
| `createdAt` | timestamp | |

**Who can read/update it:** only the user it belongs to. **Who can create it:** any signed-in user — this is intentionally permissive, since notifications are created *for* someone else (e.g. an employee's bank-change request creates a notification *for the admin*, written by the employee's own request), so the writer and the recipient are different people by design.

## 7. `/settings/company`

**Purpose:** a single document holding the company name shown in the payslip PDF header. Document ID is always the literal string `"company"` — there's only ever one of these.

| Field | Type | Notes |
|---|---|---|
| `companyName` | string | Falls back to `"PayrollPadi"` if this document doesn't exist yet — nothing breaks before an admin first sets it. |
| `updatedAt` | timestamp | |
| `updatedBy` | string | Admin UID who last saved it. |

**Who can read/write it:** admins only. Every payslip PDF generation (lock, resend, on-demand download) reads this fresh via the Admin SDK, so changing it here affects every *new* PDF from that point forward — but, consistent with the rest of the system, a payslip already locked and emailed keeps whatever name was in effect when it was generated, since the PDF itself was already created and sent.

## 8. Indexes

Firestore automatically indexes every field for simple reads **within a single collection**. It does **not** automatically index fields for `collectionGroup` queries (searching across every instance of a same-named subcollection, like `payslips`) — that has to be explicitly enabled.

This project needs exactly one such index, configured as a **single-field exemption** (Firebase Console → Firestore → Indexes → Single field → Exemptions): `payslips` collection, `employeeId` field, **Collection Group** scope, Ascending, enabled. Also declared in `firestore.indexes.json` for anyone deploying via the Firebase CLI instead of the Console.

## 9. Quick Reference — Who Can Read/Write What

| Collection | Admin read | Admin write | Employee read | Employee write |
|---|---|---|---|---|
| `/users/{uid}` | own doc only¹ | ✅ any | own doc only | ❌ never |
| `/employees/{id}` | ✅ any | ✅ any | own record only | ❌ never (goes through `bankAccountPending` via a server route instead) |
| `/taxSettings/{id}` | ✅ any | ✅ create only, never edit/delete | ❌ never | ❌ never |
| `/payrollRuns/{id}` | ✅ any | ✅ any | ❌ never (accessed only via the `payslips` subcollection) | ❌ never |
| `.../payslips/{id}` | ✅ any | ✅ any | own payslip only | ❌ never |
| `/settings/company` | ✅ any | ✅ any | ❌ never | ❌ never |
| `/notifications/{uid}/items` | own doc only¹ | ✅ any (as creator) | own items only | can create (for others), can only read/update own |

¹ Admins don't get special blanket access to *other* users' `/users` docs — this is intentional; admin privilege for employee data flows through `/employees`, not through reading other people's login-identity records.
